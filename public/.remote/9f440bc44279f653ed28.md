# これなに

- 就職活動やら転職活動でよく使われる伝家の宝刀「コミュニケーション能力」（世間的にはコミュ力とか呼ばれている）について最近振り返ってみました
- これまでマネージャーやリーダーをやってきた中で「コミュニケーションをデザイン」するにあたって何に気をつけるべきなのかをまとめました


# ⚠️ 注意点

- 以下の内容は一個人の意見です、私が所属している組織などは関係ありません
- 異論や反論認めます、むしろウェルカムです（皆さんの意見も聞いてさらに自分のスキルをブラッシュアップしていきたいので）


# コミュニケーションデザイン

## まず守るべき軸

- 相手の意見、スキル、考え方をまずは尊重しましょう
- コミュニケーション量は少な過ぎず、多過ぎずを心がけましょう
- コミュニケーションのリズムを大事にしましょう

## なぜコミュニケーションをデザインするのか

- プロジェクトを推進するにあたっては一般的には役割や責務の定義を行うべきと言われています
- 各人が各人の役割を全うするにあたっても、どのようにチームメンバー同士でコミュケーションをとってプロジェクトを進めるのかを整理した方がベターであると考えています
- 例えば開発者がコーディングするにあたっても、設計は誰がやるのか、設計の元情報はどのようにして共有するのかなどを整理する必要があります
- 上記の理由からコミュニケーションのデザインが必要だと考えています

## デザイン方法

これはあくまで自分の一例ですが、大きなステップとしては以下の5つです。

1. プロジェクトの概要を掴む
2. ステイクホルダー、チームメンバーそれぞれの性格を掴み、役割を考える
3. チームメンバーのキャリアを強くするストーリーを描き、ノイズを洗い出す
4. 必要なコミュニケーションを選別する
5. デザインしたコミュニケーションを共有し、合意を得る

それぞれの詳細を後述します。

### 1. プロジェクトの概要を掴む

当たり前の話であるかもしれませんが、プロジェクトの概要（計画、あれば設計やら開発ルールなどの内容）を掴みます。

ドメイン知識をご自身で持っていなかったり、チーム活動ルールが暗黙知になっていたりということも往々にありますが、極力理解できる範囲でふわっと理解をする努力が必要です。
この理解をしようとするアクションが無いとコミュニケーションデザインを行なってもそれをやろうという合意をとることも難しいです。
（「プロジェクトの内容わからんけど、こういう風にやろう」と人から言われても、前向きになりにくい人ももちろんいらっしゃるから）

ドキュメントがあれば大雑把でも良いから読んでいくことをおすすめします。勉強は大事。

### 2. ステイクホルダー、チームメンバーそれぞれの性格を掴み、役割を考える

チームメンバーは十人十色、スキルも性格も好き嫌いも全部違います。
コードを書きたい人がUIデザインをやれと言われたらやりがい無いし、その人のキャリア設計に影響を与えてしまいます。
このあたりは会話をしたりして掴んでいった方が良いかなと思います。

そして各メンバーの役割を整理しましょう。
組織においてはマネージャーやリーダー、ディレクターなどタイトルがついている人もいるので、そのあたりは尊重して役割を整理すると良いかなと思います。

### 3. チームメンバーのキャリアを強くするストーリーを描き、ノイズを洗い出す

1, 2番の部分は材料集め。
ここからコミュニケーションをデザインするための下ごしらえです。

チームメンバーが持つ責務ややりたいこと、性格を考慮して役割を整理します。
その時に各メンバーが単なるソルジャーとならないように、その役割にストーリーを描きます。
ストーリーは「この役割を担うことで何を達成できるのか」を描きましょう。
ただ、このストーリーはメンバーへの押し付けにならないように注意しつつ、対話を通してメンバーに納得感のあるストーリーをインストールしていきましょう。

加えて、その役割を担うにあたってノイズになるものを洗い出しておきましょう。
例えば他プロジェクトで忙しい、会議が多すぎる、わからないことがわからないという話でもOKです。
こういった情報を整理しておくことで、どのようにコミュニケーションパスを作るのかを検討する際に考慮できます。

### 4. 必要なコミュニケーションを選別する

デイリーミーティングでも良いですし、1on1でも良いです。
プロジェクトを推進し、メンバーのキャリア強化を行うにあたって必要なコミュニケーションを選別しましょう。
できるだけアドホックにならないように、チーム活動にリズムがつくような形で週単位や月単位でのデザインをしてみてください。
アドホックはチーム活動のノイズになってしまうので、定期イベントにするのが良いです。

ただし、冒頭で書きました守るべき軸にも記載しましたとおり、コミュニケーションが多過ぎるのはNGです。
毎日1on1やってますとか、目的は無いけど朝会やらデイリーミーティングをやってますというのは避けましょう。
基本的にプロジェクトを進めるためには作業が必要なので、できるだけ無駄なコミュニケーションパスは排除していきましょう。

また、コミュニケーションパスが深過ぎるのもNGです。
「本部長→部長→PdM→PM→リーダー」みたいな形でコミュニケーションが多過ぎると決定のスピードが遅くなるので、「PdM→PM」くらいのせいぜい2段階くらいまででデザインしてもらうと良いです。
どうしても決裁が必要なものがあれば決裁権者に訊くくらいの形でまとめるのが良いです。

加えて、同様に冒頭で書きました守るべき軸にもある通り、コミュニケーションが少な過ぎるのもNGです。
コミュニケーションが少な過ぎると「すみません、これってどういう話ですかね？」や「この成果物の方向性誤ってるね、ちょっと会話しようか」みたいなアドホックなコミュニケーションが増えて、各人が作業を中断して結果的にチームにストレスがかかります。

結構大変ですが、上記のような注意点を意識しつつ、必要なコミュニケーションを選別していくことが大切だと思います。

### 5. デザインしたコミュニケーションを共有し、合意を得る

コミュケーションをデザインするにあたってコミュニケーションをとるという、なんだか重複しているような感じになりますが、デザインしたコミュニケーションルールや方法、パスなどを共有してチームメンバーの合意を得ましょう。
上位下達でも良いですが、人間それぞれ思うことがあったりするので極力合意をとった方がベターです。


# コミュニケーションデザインの運用・保守

- 基本的には上記のようなデザインを行なって「運用」していきます
- ただし、チームは水物でプロジェクトフェーズやチームの成長、変化によって必要なコミュニケーションが変わっていきます
- だからこそ最初に定義したコミュニケーションに固執せず、不要なコミュニケーションがあれば落とし、リファクタした方が良いものはリファクタして「保守」していくようにしましょう


# まとめ

- プロジェクトに参画しているメンバーそれぞれが役割を全うするにあたっては、チームメンバー同士がどのようにコミュニケーションをとるのかを整理すると動きがスムーズになりますため、コミュニケーションデザインが重要だと考えています
- コミュニケーションデザインには各メンバーのキャリアを強くするストーリーを組み込むことで納得感を持ってもらいつつ、各メンバーが前向きに取り組める環境を作りましょう
- 繰り返しになりますが、あくまで一個人の意見なので、異論反論ウェルカムです（皆さんの意見も聞いてさらに自分のスキルをブラッシュアップしていきたいので）
