# はじめに

この記事は完全に私一個人の意見です。
ちゃんちゃらオカシイことも言っていると思うので、そういう部分があれば「そんなこたないしょや」というコメントいただけると幸いです。

# この記事を書こうと思った背景

最近、関わっている複数のチームで開発スピードや進め方の違いがハッキリ出てくる場面に遭遇しています。

あるチーム(Aチーム)は、4ヶ月で本番運用まで持っていきつつ複数プロダクトを並行で運営できているのに対して、別のチーム(Bチーム)は半年近く同じプロダクトを開発しているのに要件変更でなかなかリリースまでたどり着けない、という状態です。

最初は「チーム規模も違うし、扱っているドメインも違うから仕方ないのかなあ」と片付けそうになったのですが、よく観察してみるとそんな単純な話じゃないなと気づきました。むしろ AI コーディングツール(Claude Code, Cursor, Codex など)が当たり前になった今だからこそ、決定的に効いてくる要素がある気がしていて、せっかくのゴールデンウィークなのでそのあたりの考えをまとめてみます。

なお、両チームともスクラム手法を採用していて、Scrum Guide の考え方も織り交ぜつつ書いていきます。

# 両チームの差異分析

両チームの表面的な違いはいくつかあります。

- チームを取り巻く組織の規模感
- 扱っているプロダクトのドメインの複雑さ
- Developer の人数
- スクラムイベントの「重さ」(Aチームはかなりライトで Retrospective として単体のイベントはやらずに Review と一緒に Retro 的なこともやっていて、Bチームはしっかりガッツリやっています)

ただ、これらは結果に対する直接的な原因というより、もっと根っこにある要因の表れではないかと感じています。

# 仮説：Storyデザインがすべてを決めている

私が一番効いていると思っているのは、**Product Owner(PO)が描く Story の設計品質** です。

Aチームでは、

- プロダクトビジョンとロードマップが整備されている
- 「本当に必要か?」を毎回問い直すアクションが文化として根付いている
- Story が AsIs-ToBe-SoThat 構文で書かれていて、Acceptance Criteria(以降AC)が明確
- そのため Refinement はほぼ不要で、Developer は見積もり(Fibonacci)とSprint計画にすぐ入ることができている

一方Bチームでは、

- ビジョンやロードマップが曖昧
- 「Required(必須)」ではなく「Wanted(欲しい)」ベースで議論されがち
- Story の説明が「Kプロダクトと同様の機能を実装する」といったような、昔からよく言われる「ダメな要件」の典型を書きがちになっている（全員が同じ認識なら良いのですが、これまた難しい…）
- 結果として Sprint 開始後も PO と Developer の確認作業が続き、見積もりが当たらず手戻りも多い

要するに、**Developer がどれだけ頑張ろうが、上流の Story 設計が雑だと開発は前に進まない** という状態です。

# AIコーディング時代だからこそ、Storyデザインが効く

ここまでは「まあ昔から言われてる話じゃない?」と思われるかもしれません。たしかにそうです。ただ、AI コーディングツールが普通に使われる今、この差はより露骨に開発速度に跳ね返ってくる、というのが私の体感です。

Aチームでは、Story の Description をそのまま Claude Code に放り込んで、

1. Claude Code に設計とタスク整備を提案してもらう
2. Developer がレビューする
3. そのまま Claude Code に実装してもらう

という流れがかなりスムーズに回ります。これが成立しているのは、AI に渡す前の **コンテキストの品質が高い** からに他なりません。

逆に、Bチームのような Story を AI に渡しても、それっぽく動くけど的外れな実装が返ってきて、レビュー負荷がむしろ増えるだけ、という結果になりがちです。

AI コーディングツールは「仕様の質をそのまま増幅する装置」だと考えると、上流の品質がそのまま下流の生産性に直結する構造になっている。これは AI 以前から本質は同じだったのですが、AI が加速装置として機能する時代になってから、その差が「数倍」レベルから「桁違い」レベルに広がっている感覚があります。

# ただし、PO 一人のせいにすると話を間違える

ここまで読むと「結局 PO の力量がすべてなのか」と思われそうですが、そう単純化すると大事なところを見落としてしまうので、以下の3つの論点に切り分けて整理してみます。

## 論点1: Product Discovery の不足

ビジョン / ロードマップが曖昧、Required と Wanted の判断軸がないというのは、PO 個人のスキル問題というより、 **組織として Discovery にコストを払えているか** という話です。

ここで一つ補足しておきたいのが、「ドメインが複雑だからミニマム化が難しい」という言い訳は、思っているほど正しくないという点です。たとえば法令対応のような一見スコープを切りにくい領域でも、「対応範囲をどこから始めるか」「どの適用パターンを初期スコープに入れるか」を整理していけば、最小リリースは十分可能です。実際、宿泊税のような法令要件であっても、対応自治体や課税方式を絞ることで「最小で出す」整理はできます。（法令自体は違うものの、実際Aチームでもそれができていて、Bチームは全部出そうという判断をしていました）

つまり問題はドメインそのものの複雑さではなく、 **「どこを切り捨てて出すか」を判断する Discovery の精度** の方です。「ドメインが難しいからミニマム化できない」と感じている時、本当の原因は Discovery への投資不足であることが多い、というのが私の感覚です。

## 論点2: Refinement プロセスの弱さ

Scrum Guide(2020) では、Refinement は Developer も主体的に関わるイベントとされています。つまり Story が曖昧なら、Sprint Planning に乗る前に Developer 側から差し戻すのが本来の運用です。

「Story が Ready の状態で Sprint に入る」というガードがチームに無いまま Sprint を回すと、当然ながら Sprint 開始後に確認作業が発生する。これは PO だけの問題ではなく、**チーム全体として Ready の基準を持っていない** 問題です。

具体的に打てる手としては

- Refinement で AC の形式を AsIs-ToBe-SoThat に揃える運用を提案する
- Definition of Ready(DoR)をチームで明文化する(Scrum Guide では必須ではないが、運用補助としては有効)
- 「〇〇と同様の機能」のような曖昧な Story は Sprint Planning で差し戻すルールにする

といったことになるかなと考えています。Scrum Master 的な立場で介入するなら正当化しやすい領域かと。

## 論点3: チームサイズと認知負荷

Developer の人数が増えると、コミュニケーション経路の数が線形ではなく組み合わせ的に増えます(2人なら1本、4人なら6本)。共通理解を作るコスト、レビュー待ち行列、コードベースの認知負荷、すべてに効いてきます。

なので、Story の品質を改善しても「人数が多いチーム特有の摩擦」は残ります。これも構造的な制約として認識しておかないと、改善しても効果が出ないと感じてフラストレーションが溜まる原因になります。

# AIコーディング時代の組織への投資という視点

ここまでをまとめると、AI コーディングツールを最大限活用したいなら、

1. **Discovery にちゃんとコストを払う**(ビジョン / ロードマップ / Required の判断軸)
2. **Refinement プロセスを強くする**(Ready でない Story を Sprint に入れない)
3. **構造的な制約を認識する**(ドメイン特性、チームサイズ)

という3層で投資を考える必要があります。

そして重要なのは、**1 と 2 を整備することは「AI がレバレッジ可能な仕様」を整備することとほぼ同義** だということ。逆に言えば、ここを整備しないまま Claude Code や Cursor を導入しても、期待したほどの効果は出ません。

「AI ツールを入れる前に、まず Story の書き方を整える」というのは、地味だけど提案としても通りやすい論点になるんじゃないかと思っています。AI ツール自体のライセンス費用より、PO や Discovery プロセスへの投資の方がよほど ROI が高いケースは多いはずです。

# おわりに

「PO の Story 設計がボトルネック」という最初の仮説自体は間違っていないと思っていますが、それを単一原因として語ると PO 個人への批判に聞こえてしまいます。なので

- Discovery の不足(組織課題)
- Refinement プロセスの弱さ(チーム課題)
- ドメイン特性とチームサイズ(構造制約)

の3層に切り分けて、それぞれに合った打ち手を提案するのが現実的ではなかろうかと考えている今年のゴールデンウィークです。

AI コーディングツールが普通になった時代だからこそ、上流の設計品質が桁違いに効いてくる。逆に言えば、上流が整っているチームにとっては、AI は本当に強力な加速装置になります。今いる現場で、自分が打てる小さな改善から積み重ねていきたいなと思います。
