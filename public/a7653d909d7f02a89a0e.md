# これなに

スクラム開発をしていると、バックログにこんなチケットが並んでいることはないでしょうか。

- 「ログイン機能を実装する」
- 「DBにユーザーテーブルを追加する」
- 「エラーハンドリングを追加する」

一見、具体的でわかりやすそうですよね。でもこれ、**ユーザーストーリーではなくタスクチケット**になっています。

この記事では、なぜチームのストーリーチケットがタスクベースに陥りがちなのかを整理した上で、ユーザー価値の視点でストーリーを書くために自分が意識しているコツというかテクニック的なものを書こうと思います。


# なぜあなたのストーリーチケットはタスクベースになってしまうのか

タスクベースのストーリーが生まれてしまうのはDevelopersチームがそれをできないからタスクベースの指示してしまうというわけではないです。
自分も同様な経験があったので、その頃の自分の考えを振り返りながらよくある原因を列記します。

### 原因1：実装視点から書き始めてしまう

「次のスプリントで何をやるか」を議論するとき、自然と「何を作るか」から思考します。その結果、チケットには「〇〇を実装する」「〇〇APIを叩く処理を追加する」といった、実装そのものの記述が入ってしまいます。

### 原因2：Scrum Guide がストーリーの書き方を規定していない

実は Scrum Guide には「ユーザーストーリー」という言葉は一切登場しません。つまり、明示的なフォーマットが存在しないがゆえになんとなくタスクを書いてしまい、結果的にタスクチケットになりがちになってしまいます。（実際初めてスクラムに取り組んだ時はこの落とし穴に何度もハマりました）

### 原因3：「誰のために作るか」が共有されていない

ユーザーストーリーの核心は「誰の、どんな問題を解決するか」です。チーム内でユーザーペルソナの解像度が低いと、ストーリーを書く際に思わず実装の話にすり替わってしまいます。


# そもそもなぜユーザー価値視点でユーザーストーリーを書くのか

[Atlassianが公開しているユーザーストーリー作成ガイド](https://www.atlassian.com/agile/project-management/user-stories)には、ユーザーストーリーの目的がこう説明されています。

> ユーザーストーリーを読んだチームは「なぜ作るのか」「何を作るのか」「それがどんな価値を生み出すのか」を理解できる。

つまりユーザーストーリーとは、**「なぜ（Why）」をチームが共有するための道具**です。

タスクベースのチケットは「何を作るか（What）」はわかりますが、「なぜ作るか（Why）」が抜け落ちています。この状態だと、次のような問題が起きやすくなります。

- **チームが目的を見失う**：作業を消化することが目的になり、ユーザーへの価値提供という本来の目的がぼやける
- **判断ができなくなる**：開発中に仕様の曖昧な点が出てきたとき、「ユーザーにとって何がベストか」という基準がないため、意思決定が止まる
- **スプリントゴールと紐づかなくなる**：Scrum Guideが定める「なぜこのスプリントは価値があるのか（スプリントゴール）」とタスクが繋がらず、バラバラな作業の寄せ集めになる

ユーザー価値を起点にストーリーを書くことで、チームは「何を作るか」だけでなく「誰のために、なぜ作るか」を共有した状態で開発に臨むことができます。


# ユーザー価値視点でユーザーストーリーを書く際に意識すべきこと

### 基本テンプレートを使う

個人的には Atlassian が公開しているフォーマットがシンプルで利用しやすいのでおすすめです。
紹介すると以下のようなフォーマットです：

```
As a [ペルソナ], I want to [やりたいこと], so that [その理由・得られる価値].
```

日本語に置き換えるなら

```
[ペルソナ] として、[やりたいこと] がしたい。なぜなら [得られる価値・解決される問題] だから。
```

**例：**
- ❌ タスクベース：「パスワードリセット機能を実装する」
- ✅ ユーザー価値ベース：「ユーザーとして、パスワードを忘れてもリセットできるようにしたい。なぜなら、ログインできなくなる不安なく安心してサービスを使い続けられるから。」

### コツ1：「UIや実装の話が入ったら一度止める」

ストーリーを書く途中で「ボタンを追加する」「APIを呼び出す」などの実装の話が出てきたら、それはWhatやHowの話です。まず「誰が・なぜ・何を得たいのか」に立ち返りましょう。

### コツ2：「so that（なぜなら）」を必ず書く

タスクベースになっているストーリーの多くは、`so that` の部分が省略されています。ここを埋めることで、チームがユーザーの目的を常に意識できるようになります。

**`so that` を書くことで何が変わるか：**

| 省略した場合 | `so that` を書いた場合 |
|---|---|
| 実装が完了したら「完了」になる | ユーザーが価値を得られたら「完了」になる |
| チームは作業を消化する | チームはユーザーの問題を解決する |

### コツ3：ペルソナの解像度を上げる

「ユーザーとして」という書き方は便利ですが、ペルソナが曖昧だとストーリーも曖昧になります。可能な限り具体的なペルソナ名（例：「初めて使う一般ユーザーのAさん」「毎日レポートを確認する管理者のBさん」）を使うことで、チームのイメージが統一されます。

### コツ4：1スプリントで完了できるサイズに分割する

ストーリーが大きすぎると、スプリント内に収まらないだけでなく、タスクベースに分解されやすくなります。1スプリントで「ユーザーが価値を体験できる」単位に分割することを意識しましょう。大きすぎるものはエピックとして扱い、ストーリーに落とし込みます。

### コツ5：完成の定義（DoD）をセットで決める

ストーリーを書いたら、「このストーリーはいつ完了か」をチームで合意します。Scrum Guideのインクリメントにおける「完成の定義（Definition of Done）」の考え方と連動させると、「ユーザーが目標を達成できる状態になったとき」を完了の基準にしやすくなります。


# まとめ

タスクベースのチケットが生まれるのは「Why（なぜ）を書く習慣がないこと」と「ユーザーペルソナの解像度が低いこと」が主な原因かなと考えています。
ユーザーストーリーを価値ベースで書くために押さえておきたいポイントを改めて以下の通りまとめます。

| チェックポイント | 確認内容 |
|---|---|
| テンプレート | `As a / I want to / so that` の形式になっているか |
| ペルソナ | 「ユーザー」ではなく具体的な人物・役割が書かれているか |
| Why | `so that` で得られる価値・解決される問題が書かれているか |
| 実装の話 | UIや技術的な話がストーリー本文に入っていないか |
| サイズ | 1スプリントで完了できる粒度になっているか |
| 完了の定義 | ユーザーが価値を得た状態を「完了」として定義しているか |

Scrum Guideは「何をどう書くか」を規定しません。だからこそ、チームが意識的にユーザー価値の視点を持ち続けることが、プロダクトの質を左右します。まずは次のスプリントのバックログを見直して、`so that` が書かれていないストーリーを一つ書き直してみるところから始めてみましょう。


# 最後に

いろいろと説教くさいことを書きましたが、未来の自分が振り返る時にも使えるようにしたいなと思い記事を書きました。
あとはいつ何時も絶対にユーザー価値に依って書く必要はないかなと自分は考えていまして、考え抜いた結果最終的にタスクベースに落ちるのは一定仕方ない部分はあるかなと思います。（例えば「リポジトリ作成」とかはもはやユーザー価値を書く必要もないので、そういうのは仕方ない）
ただ、ストーリー作成時の意識の中にユーザー価値を考えているかどうかは大きな差に繋がってくるので、ぜひ上記のポイントは意識してみてください！
