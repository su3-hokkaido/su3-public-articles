## これなに

PHP 8.3 => 8.4 にアップグレードした際、サーバが `404 Not Found` を返し続けて開けなくなったので、ClaudeCode と一緒に対応した時の反省記録。

## 環境

- **アプリケーション**: Laravel 13.6
- **PHP**: 8.3.14 → 8.4.20 へのアップグレード
- **インフラ**: Ubuntu 22.04 (jammy) + nginx 1.18 + PHP-FPM
- **デプロイ**: Capistrano/Envoyer 系（`releases/{N}/` + `current` symlink + `shared/`）
- **マルチテナント**: `{tenant}.su3-hokkaido.co` 形式のサブドメイン構成
- **PHP-FPM プロセスの実行ユーザー**: `su3-hokkaido`（アプリ所有者）
- **nginx worker user**: `su3-hokkaido`（一般的な `www-data` ではない、運用都合のカスタム）

## 発生したこと

ブラウザでアクセスすると、Laravel の例外ハンドラから JSON 404 が返ってきた。

```json
{
    "message": "No query results for model [App\\Models\\Tenant]."
}
```

その後、nginx を再起動したら、今度は nginx 自身が 404 を返すようになった。

```bash
$ curl -i https://su3-hokkaido.co/admin/login
HTTP/1.1 404 Not Found
Server: nginx/1.18.0 (Ubuntu)
Content-Type: text/html
Content-Length: 162

<html>
<head><title>404 Not Found</title></head>
<body>
<center><h1>404 Not Found</h1></center>
<hr><center>nginx/1.18.0 (Ubuntu)</center>
</body>
</html>
```

`Content-Length: 162` のいかにもなnginxデフォルト 404 ページ。Laravel まで届いていないようでした。

## 問題発生前後での条件

今回の問題にさしあたって、状況を整理すると

1. composer の依存パッケージ更新で `phpunit ^13` と `laravel/framework ^13.6` を取り込んだ
2. その依存が PHP 8.4 を要求するため、サーバの PHP を 8.3 → 8.4 にアップグレードする作業をしていた
3. 作業ガイドを書いてからその通りに進めていたつもり

## 調査の記録

### Step 1: まず Laravel ログと nginx エラーログを確認

最初に出ていた `No query results for model [App\Models\Tenant]` は、サブドメインから tenant code を引いて `Tenant::where('code', $code)->firstOrFail()` する箇所で出るので、DB にレコードはあるはず。

まず Laravel ログと nginx エラーログを並べて見ると、Laravel ログ側は無風なのに nginx 側に次のようなログが大量に出ていました。

```bash
$ sudo tail -n 30 /var/log/nginx/error.log
2026/04/27 11:23:21 [error] FastCGI sent in stderr:
"PHP message: PHP Fatal error:
Composer detected issues in your platform:
Your Composer dependencies require a PHP version \">= 8.4.0\".
You are running 8.3.14.
in /var/www/su3-hokkaido/shared/vendor/composer/platform_check.php on line 22"
upstream: "fastcgi://unix:/run/php/php8.3-fpm.sock:"
```

→ ⚠️ `php8.3-fpm.sock` に向かってリクエストが流れ続けていて、PHP 8.3 環境で composer の platform_check に弾かれている。

つまり nginx の `fastcgi_pass` が **古いソケットを参照しっぱなし**だったというのが第一原因。

### Step 2: nginx の `fastcgi_pass` を 8.4 に向ける

```bash
$ sudo grep -rln 'php8\.3-fpm\.sock' /etc/nginx/
/etc/nginx/sites-available/default

$ sudo sed -i.bak 's|php8\.3-fpm\.sock|php8.4-fpm.sock|g' \
    $(sudo grep -rl 'php8\.3-fpm\.sock' /etc/nginx/)

$ sudo nginx -t
nginx: configuration file /etc/nginx/nginx.conf test is successful

$ sudo systemctl reload nginx
```

これで直るはずと思って curl したら、まだ 404 のまま。

```bash
$ curl -I https://su3-hokkaido.co/admin/login
HTTP/1.1 404 Not Found
Server: nginx/1.18.0 (Ubuntu)
```

→ ❌ 直っていない...

### Step 3: もう一度 nginx エラーログを確認

```bash
$ sudo systemctl restart php8.4-fpm
$ sudo systemctl restart nginx
$ sudo truncate -s 0 /var/log/nginx/error.log
$ curl -i https://su3-hokkaido.co/admin/login
$ sudo tail -n 30 /var/log/nginx/error.log

2026/04/27 15:37:07 [crit] connect() to unix:/run/php/php8.4-fpm.sock failed
  (13: Permission denied) while connecting to upstream
2026/04/27 15:37:07 [error] open() "/usr/share/nginx/html/50x.html" failed
  (2: No such file or directory)
```

→ 📌 `Permission denied` でソケットに接続できていない。

しかも次の行で **`/usr/share/nginx/html/50x.html` が無い**と言っている。（この話は別で後述）

### Step 4: ソケットの所有者を調べる

```bash
$ ls -la /run/php/php8.4-fpm.sock
srw-rw---- 1 www-data www-data 0 Apr 27 15:25 /run/php/php8.4-fpm.sock
```

→ ソケットの所有者・グループが **`www-data:www-data`** で、モードは `0660`。

ところが、nginx のワーカーは

```bash
$ ps -eo user,comm | grep nginx | grep -v grep
root     nginx
su3-hokkaido nginx
su3-hokkaido nginx
su3-hokkaido nginx

$ sudo grep "^user" /etc/nginx/nginx.conf
user su3-hokkaido;
```

→ nginx は `su3-hokkaido` ユーザーで動いているのに、ソケットは `www-data:www-data` の 660。**`su3-hokkaido` はソケットに接続できない**。

### Step 5: PHP 8.3 時代の pool 設定と比較

```bash
$ grep -E "^(user|group|listen\.(owner|group|mode))\b" /etc/php/8.3/fpm/pool.d/www.conf
user = su3-hokkaido
group = su3-hokkaido
listen.owner = su3-hokkaido
listen.group = su3-hokkaido

$ grep -E "^(user|group|listen\.(owner|group|mode))\b" /etc/php/8.4/fpm/pool.d/www.conf
user = www-data
group = www-data
listen.owner = www-data
listen.group = www-data
```

→ ✅ 原因確定。

PHP 8.3 では運用上 `su3-hokkaido` で動かすカスタムをしていたのに、PHP 8.4 を `apt install` した直後は **ディストリ標準のデフォルト設定（`www-data`）で pool が生成される**ため、設定差分が発生していたということが判明。

### Step 6: pool 設定を 8.3 から移植

```bash
$ sudo sed -i.bak \
    -e 's|^user = .*|user = su3-hokkaido|' \
    -e 's|^group = .*|group = su3-hokkaido|' \
    -e 's|^listen\.owner = .*|listen.owner = su3-hokkaido|' \
    -e 's|^listen\.group = .*|listen.group = su3-hokkaido|' \
    /etc/php/8.4/fpm/pool.d/www.conf

$ sudo systemctl restart php8.4-fpm

$ ls -la /run/php/php8.4-fpm.sock
srw-rw---- 1 su3-hokkaido su3-hokkaido 0 Apr 27 15:41 /run/php/php8.4-fpm.sock
```

→ ✅ ソケットの所有者が `su3-hokkaido:su3-hokkaido` になった。

```bash
$ curl -i https://su3-hokkaido.co/admin/login | head -10
HTTP/1.1 200 OK
Server: nginx/1.18.0 (Ubuntu)
Content-Type: text/html; charset=utf-8
Set-Cookie: XSRF-TOKEN=...
...
<title>su3-hokkaido Admin</title>
```

→ ✅ 復旧。ログイン画面の HTML がちゃんと返ってきた。

## なぜ最初の症状が「Tenant が見つからない」だったのか

後から振り返って整理した経緯はこんな感じ。

```
[1] composer install で vendor が更新（PHP 8.4 必須に）
    ↓
[2] PHP 8.4 はインストール済みだが、nginx の fastcgi_pass は php8.3-fpm.sock のまま
    ↓
[3] 一部のリクエストは PHP 8.3 で処理 → composer/platform_check.php が Fatal error
    ↓
[4] その Fatal error がレスポンスとして整形される過程で、
    Laravel の例外ハンドラまで部分的に動いてしまい、
    たまたま近くで動いていた Tenant 解決ロジックの ModelNotFoundException が
    JSON 404 として表面化した（推測）
    ↓
[5] ユーザー目線では「Tenant 404」に見えるが、実態は PHP version mismatch
```

つまり、**Laravel の 404 メッセージが、本当の原因（インフラ層の version mismatch）を覆い隠していた**形でした。エラーメッセージが直感的に間違った方向を指している、という現象。

## なぜ nginx が 502 ではなく 404 を返したか

この話も引っかかったポイント。
普通、PHP-FPM ソケットに繋がらなければ 502 Bad Gateway が返るはずです。それが 404 になっていた理由は nginx 設定にありました。

```nginx
server {
    listen 443 ssl;
    server_name su3-hokkaido.co;
    
    error_page 500 502 503 504 /50x.html;
    location = /50x.html {
        root /usr/share/nginx/html;
    }
    ...
}
```

`error_page` ディレクティブで 502 を `/50x.html` にフォールバックさせていますが、**`/usr/share/nginx/html/50x.html` 自体が存在していない**ため、最終的に nginx は 404 を返していました。

```bash
$ ls -la /usr/share/nginx/html/50x.html
ls: cannot access '/usr/share/nginx/html/50x.html': No such file or directory
```

この話、本来 502 が出ていれば upstream に問題があるとわかるところを、**404 として現れたために Laravel のルーティング問題と区別がつかない**まま時間を溶かした形。（Step 3 でエラーログをちゃんと読めばよかった）

## 切り分けで使った検証コマンド集

同じトラブルに当たった人が再利用できるよう、本件で使った検証コマンドをまとめておきます。

### nginx ↔ PHP-FPM 接続の整合性

```bash
# nginx worker user
ps -eo user,comm | grep nginx | grep -v grep
sudo grep '^user' /etc/nginx/nginx.conf

# PHP-FPM ソケット所有者・モード
ls -la /run/php/

# PHP-FPM プール設定
grep -E "^(user|group|listen\.(owner|group|mode))\b" \
    /etc/php/8.4/fpm/pool.d/www.conf

# nginx の fastcgi_pass 参照先
sudo nginx -T 2>&1 | grep fastcgi_pass
```

`nginx worker user`、`ソケット所有者`、`pool の listen.owner` が**全部一致している**ことが必須。どれか 1 つズレると `Permission denied`。

### エラーログをクリアして実リクエスト

切り分け中はノイズが多いので、ログをクリアしてから 1 リクエスト打って観察するのがおすすめ。

```bash
sudo truncate -s 0 /var/log/nginx/error.log
curl -i https://your-domain/path
sudo tail -n 30 /var/log/nginx/error.log
```

### 502 / 404 の真因を見分ける

| nginx error.log のメッセージ | 真因 |
|---|---|
| `connect() ... sock failed (13: Permission denied)` | ソケット所有者・モードと nginx user の不整合（pool 設定移植漏れ） |
| `connect() ... sock failed (2: No such file or directory)` | ソケットが存在しない（PHP-FPM 停止 or `fastcgi_pass` のソケット名間違い） |
| `PHP message: PHP Fatal error: Composer detected issues in your platform` | `fastcgi_pass` が古い PHP バージョンのソケットを参照している |
| `open() "/usr/share/nginx/html/50x.html" failed` | `error_page` のフォールバック先が無いため 502 → 404 になっている |

## 今回の教訓

### 1. PHP メジャー/マイナーアップグレード時は pool 設定の移植をするべき

`apt install php8.4-fpm` でディストリのデフォルト設定が降ってきます。本番運用していると pool 設定（特に `user`/`group`/`listen.owner`/`listen.group`）はカスタムされていることが多いので、**8.3 と 8.4 で `diff` を取って必ず差分を移植したほうがベター**。

```bash
sudo diff /etc/php/8.3/fpm/pool.d/www.conf /etc/php/8.4/fpm/pool.d/www.conf
```

### 2. `fastcgi_pass` のソケット名はバージョン入りなのでアップグレード時に必ず変わる

`/run/php/php8.X-fpm.sock` というパスにバージョン番号が含まれているので、PHP マイナー or メジャーアップグレード時は **必ず nginx 側も書き換える**必要があります。逆に言うと、これを書き換えない限り旧 PHP が動いている限りは見かけ上動く。（だからこそ気付きにくい…）

### 3. `error_page` のフォールバック先は実在を確認しておく

```nginx
error_page 500 502 503 504 /50x.html;
location = /50x.html {
    root /usr/share/nginx/html;
}
```

このパターンの設定を持っているなら、`/usr/share/nginx/html/50x.html` が**実在することを確認**するか、最低限のメンテナンスページを置いておきましょう。本来 502 で返るべき障害が 404 として現れると、切り分けに使うエラーコードの情報量がガクッと落ちます。

```bash
sudo tee /usr/share/nginx/html/50x.html >/dev/null <<'EOF'
<!DOCTYPE html>
<html lang="ja">
<head><meta charset="utf-8"><title>Service Unavailable</title></head>
<body><h1>503 Service Temporarily Unavailable</h1></body>
</html>
EOF
```

### 4. Laravel のエラーメッセージを鵜呑みにしない

「`No query results for model [Tenant]`」と出ていたら普通は DB / モデル周りを疑いますが、今回は **PHP version mismatch によって例外ハンドラが部分的に動いた結果出てしまった偽情報**でした。

エラーメッセージは「最後に表に出てきた一番上っ面の情報」であって、必ずしも root cause ではない。インフラレイヤで何かが崩れている時はこういう偽情報が出やすいので、**怪しいときは Laravel ログだけでなく nginx error.log や PHP-FPM ログまで縦横断で見る**ことが大事。

### 5. 作業ガイドは「最重要ステップを冒頭で目立たせる」

今回、自分で書いた作業ガイドの通り進めていたはずなのに、Step 5（nginx の `fastcgi_pass` 切替）と pool 設定の移植が抜けていました。文章中に淡々と書かれていると、視線がスっと滑って飛ばしてしまうらしい。

対策として、ガイドの冒頭に「**ここを飛ばすと事故になる**」と明示した重要ステップ一覧と、過去の事故記録を載せるよう書き直しました。**未来の自分（or チームメンバー）への警告として「ここで具体的にこうハマった」を残す**のが効くと思っています。

## 余談：ClaudeCode にめちゃくちゃ伴奏してもらった

例によって ClaudeCode に伴走してもらいました。
特に効いたのは nginx error.log の `Permission denied` に関して、ソケット所有者と nginx worker user の不一致を即座に指摘してくれた点。自分が持っている知識だけだと「FPM 起動してるしソケットもあるのに何で？」で詰まっていたかもしれない…本当に勉強になる。

## まとめ

| 項目 | 内容 |
|---|---|
| **症状（第1段階）** | Laravel から `No query results for model [Tenant]` の JSON 404 |
| **症状（第2段階）** | nginx 再起動後、nginx 自身が静的に 404 を返す |
| **直接原因（第1段階）** | nginx の `fastcgi_pass` が `php8.3-fpm.sock` のままで、PHP 8.3 で composer/platform_check が Fatal error |
| **直接原因（第2段階）** | `fastcgi_pass` を 8.4 に直したが、PHP 8.4 の pool 設定がデフォルト `www-data` のため nginx (`su3-hokkaido`) からソケットに `Permission denied` |
| **根本原因** | PHP メジャー/マイナーアップグレード時に「`fastcgi_pass` のソケット名」と「PHP-FPM pool 設定（`user/group/listen.*`）」の移植漏れ |
| **404 になった補助要因** | `error_page 502 ... /50x.html;` で 50x.html が実在せず、502 が 404 として表面化 |
| **対応** | pool 設定を 8.3 から `sed` で移植、`fastcgi_pass` を 8.4 ソケットに変更、`php8.4-fpm` と `nginx` を restart |
| **教訓** | PHP メジャー/マイナーアップグレードは「PHP を入れる」だけでは終わらない。ディストリのデフォルト設定で上書きされる pool 設定の差分移植までセットで考える |

似たような構成で運用しているチームの参考になれば幸いです！
